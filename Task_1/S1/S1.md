## 🧠 The Query — Explained in Plain English

The query is written using something called a **CTE (Common Table Expression)**. Think of a CTE like building blocks — you create one piece at a time and stack them on top of each other.

There are **3 parts** to this query.

---

### Part 1 — Add Up Sales Per Year

**What it does in plain English:**
> "Look at every row in the Sales table. Group them by year. Add up all the amounts for each year."

It is like sorting a pile of receipts by year, then counting the total for each pile.

```sql
WITH YearlySales AS (
  SELECT
    YEAR(SaleDate)  AS SalesYear,
    SUM(Amount)     AS TotalSales
  FROM Sales
  GROUP BY YEAR(SaleDate)
),
```

| Keyword | What it does |
|---------|-------------|
| `YEAR(SaleDate)` | Pulls just the year from a full date. E.g. `2022-05-22` becomes `2022` |
| `SUM(Amount)` | Adds up all Amount values for that year |
| `GROUP BY YEAR(SaleDate)` | Makes one row per unique year |
| `AS YearlySales` | Names this block so we can refer to it later |

---

### Part 2 — Look Up the Previous Year's Sales

**What it does in plain English:**
> "For each year, also fetch what the sales were in the year before — so we can compare them."

This uses a special function called `LAG()`. Think of it like looking over your shoulder at the row behind you.

```sql
SalesWithGrowth AS (
  SELECT
    SalesYear,
    TotalSales,
    LAG(TotalSales) OVER (ORDER BY SalesYear) AS PrevYearSales
  FROM YearlySales
)
```

| Keyword | What it does |
|---------|-------------|
| `LAG(TotalSales)` | Gets the `TotalSales` value from the previous row |
| `OVER (ORDER BY SalesYear)` | Tells LAG to look at rows sorted by year |
| `AS PrevYearSales` | Names the result so we can use it in Part 3 |

> 💡 For the very first year (2021), there is no row before it — so `PrevYearSales` will be **NULL**. That is expected.

---

### Part 3 — Calculate the Growth Percentage

**What it does in plain English:**
> "Now that I have this year's total AND last year's total — calculate how much it grew as a percentage."

The formula is:

```
Percent Growth = ( (This Year - Last Year) / Last Year ) × 100
```

```sql
SELECT
  SalesYear,
  TotalSales,
  ROUND(((TotalSales - PrevYearSales) * 100.0 / PrevYearSales), 2)
    AS PercentGrowth
FROM SalesWithGrowth;
```

| Keyword | What it does |
|---------|-------------|
| `TotalSales - PrevYearSales` | Finds the difference between this year and last year |
| `* 100.0` | Converts to percentage. The `.0` forces decimal division (not whole number) |
| `/ PrevYearSales` | Divides by last year to get the ratio |
| `ROUND(..., 2)` | Rounds to 2 decimal places. E.g. `20.001` becomes `20.00` |

---

## 📋 Full Query — Copy This Into SSMS

Paste this entire block into a **new query window** and press **F5** to run it.

```sql
WITH YearlySales AS (
  SELECT
    YEAR(SaleDate) AS SalesYear,
    SUM(Amount)    AS TotalSales
  FROM Sales
  GROUP BY YEAR(SaleDate)
),
SalesWithGrowth AS (
  SELECT
    SalesYear,
    TotalSales,
    LAG(TotalSales) OVER (ORDER BY SalesYear) AS PrevYearSales
  FROM YearlySales
)
SELECT
  SalesYear,
  TotalSales,
  ROUND(((TotalSales - PrevYearSales) * 100.0 / PrevYearSales), 2)
    AS PercentGrowth
FROM SalesWithGrowth;
```

## ✅ Validation — How Your Work is Checked

Once you submit your CSV file, an automated validator compares it against the correct answer file. Here is the **exact logic** the validator uses:

---

### What the Validator Checks

```
1. Sort both files by SalesYear (ascending) before comparing
2. For each row, compare: SalesYear | TotalSales | PercentGrowth
3. Apply tolerance rules (see below) before marking pass or fail
```

---

### Tolerance Rules — "Near But Correct" Still Passes ✅

The validator is designed to be **forgiving on format, strict on logic**. If your SQL logic is correct but your output has minor formatting differences, you will still pass.

| Situation | Example | Verdict |
|-----------|---------|---------|
| Exact match | `20.00` vs `20.00` | ✅ Pass |
| Different decimal places | `20.0` vs `20.00` | ✅ Pass — rounded to 2dp before compare |
| Extra decimal digits | `20.001` vs `20.00` | ✅ Pass — rounded to 2dp before compare |
| NULL vs empty string | `NULL` vs ` ` | ✅ Pass — both treated as "no value" |
| NULL vs 0 | `NULL` vs `0` | ❌ Fail — NULL means unknown, 0 means zero growth |
| Row order different | 2024 before 2021 | ✅ Pass — validator sorts both before comparing |
| Column names different | `PercentGrowth` vs `pct_growth` | ✅ Pass — compared by position, not name |
| Wrong growth value | `20.00` vs `25.00` | ❌ Fail — incorrect logic |
| Missing year row | Only 3 rows instead of 4 | ❌ Fail — all 4 years must be present |
| Extra rows | 5 rows instead of 4 | ❌ Fail — data may have been inserted twice |

---

### Validation Logic (Pseudocode)

This is exactly how the comparison function works behind the scenes:

```python
def validate(user_csv, answer_csv):

    user    = sort_by_year( load(user_csv) )
    answer  = sort_by_year( load(answer_csv) )

    # Check row count
    if len(user) != len(answer):
        return FAIL, "Wrong number of rows"

    for i in range(len(answer)):
        u = user[i]
        a = answer[i]

        # Check SalesYear (must be exact)
        if u["SalesYear"] != a["SalesYear"]:
            return FAIL, f"Wrong year at row {i+1}"

        # Check TotalSales (rounded to 2dp)
        if round(u["TotalSales"], 2) != round(a["TotalSales"], 2):
            return FAIL, f"Wrong TotalSales at year {a['SalesYear']}"

        # Check PercentGrowth (NULL-safe, rounded to 2dp)
        u_growth = None if is_null(u["PercentGrowth"]) else round(u["PercentGrowth"], 2)
        a_growth = None if is_null(a["PercentGrowth"]) else round(a["PercentGrowth"], 2)

        if u_growth != a_growth:
            return FAIL, f"Wrong PercentGrowth at year {a['SalesYear']}"

    return PASS, "Task completed successfully!"
```

---

### Why This Design Works

The dataset uses **round numbers** that produce **clean percentages**:

```
2021 → 10,000   (base year)
2022 → 12,000   → growth = (12000 - 10000) / 10000 × 100 = 20.00%  ✅ exact
2023 → 15,000   → growth = (15000 - 12000) / 12000 × 100 = 25.00%  ✅ exact
2024 → 18,000   → growth = (18000 - 15000) / 15000 × 100 = 20.00%  ✅ exact
```

This means even if a user rounds slightly differently (e.g. to 1dp instead of 2dp), the rounded-to-2dp comparison still matches. There are no tricky floating-point values like `19.9999...` that could cause false failures.

## 🔴 Common Mistakes and How to Fix Them

| Problem | Most Likely Cause | Fix |
|---------|-------------------|-----|
| `"Invalid object name 'Sales'"` | Table not created yet | Run the `CREATE TABLE` + `INSERT` steps first |
| Growth shows as `0` for all years | Integer division | Make sure you have `* 100.0` not `* 100` |
| First year shows `0` growth instead of `NULL` | Wrong NULL handling | Use `LAG()` — do not manually set it to 0 |
| CSV has no header row | Export setting | Re-export and ensure headers are included |
| 16 rows in results instead of 4 | Forgot `GROUP BY` | Check that `GROUP BY YEAR(SaleDate)` is present |
| Validator says "wrong number of rows" | Data inserted twice | Run `DELETE FROM Sales` then re-run the INSERT block |
| Wrong database selected | Query ran on wrong DB | Check the dropdown at the top of the SSMS toolbar |
